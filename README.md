# check-app プロジェクト完全ガイド

このドキュメントは、今回のプロジェクトを 要件定義 から 実装・公開 まで一気通貫で理解するための手引きです。
対象読者は、アプリ開発のワークフローが初めての方です。

## 0. このプロジェクトで作ったもの

CSVを読み込み、1行ずつカード表示で確認し、次の操作ができるWebアプリです。

- いいね（ハート）のON/OFF切り替え
- カード単位のコメント追加・削除
- CSVファイルアップロードとCSV平文貼り付けの両対応
- いいねされた行のみを新しいCSVとしてエクスポート

スマートフォン前提のUIにしており、GitHub Pages公開で同一Wi-Fi不要で使える設計です。

## 1. 要件定義（何を決めるべきか）

要件定義は、作る前にゴールと制約を明文化する工程です。
初心者ほど、ここを丁寧にやると実装の迷いが激減します。

### 1-1. 要件定義で最低限決める項目

- 誰が使うか: 単一ユーザーか、複数ユーザーか
- どこで使うか: ローカル限定か、インターネット公開か
- 入力データ形式: CSVのヘッダー有無、文字コード、想定列
- 操作対象の粒度: ファイル単位・行単位・セル単位
- 完了条件: 何を満たしたらエクスポート可能か
- 出力仕様: どの行を出すか、追加列の名前と型
- 非機能要件: スマホ操作性、レスポンス、可読性、保守性

### 1-2. 今回の確定要件

- 利用者: 単一ユーザー
- 画面: スマホ優先、PCでも利用可能
- 入力: CSVファイルアップロードまたはCSV平文貼り付け、ヘッダーは1行目を採用
- 表示: 1ページ1行のカード表示
- いいね: カード単位でON/OFF可能
- コメント: カード単位で複数コメント
- 完了判定: 最後の行に一度でも到達したらエクスポート可能
- 出力: いいねONの行のみ抽出し、追加列を付与

### 1-3. 要件定義のTips

- Tips 1: UIの粒度と出力粒度を分けて考える
	例: 操作はカード単位、出力は行単位。
- Tips 2: 完了判定は曖昧にしない
	例: 「最後の行に到達」を条件に固定し、ユーザー操作の解釈差をなくす。
- Tips 3: 出力列の型を先に決める
	例: commentsはJSON文字列に統一。
- Tips 4: 初期は単一ユーザーに絞る
	認証・同期を外すだけで実装難易度が大幅に下がる。

## 2. 設計（実装前の構造化）

### 2-1. 画面設計

- ヒーロー領域: アプリ説明
- 入力領域: CSVアップロード + CSV平文貼り付け
- 状態領域: 総行数、最終行到達状態、いいね対象数、ダウンロードボタン
- カード領域: 1ページ1カード
	- 上部: カード番号
	- 中央: 業界・事例名（小さく）、説明（メイン）、専門用語解説（補助）
	- 下部: コメント欄
	- その下: いいねトグル
- 最下部: ページ遷移

### 2-2. データ設計

内部状態は、次のようなキーで管理します。

- likedRows: 行番号 -> true
- reachedLastRow: 最終行に到達済みかどうか
- comments: 行番号 -> コメント配列

### 2-3. 出力CSV設計

エクスポート対象は いいねON の行のみ。
出力列は次の通りです。

- 元CSVの全列
- liked
- comments

comments列は JSON文字列を保存します。

## 3. 実装（このプロジェクトで何をしたか）

### 3-1. 技術スタック

- フロントエンド: React + Vite
- CSV処理: Papa Parse
- 開発補助: nodemon, concurrently
- 公開: GitHub Pages + GitHub Actions

### 3-2. 主要ファイル

- [src/App.jsx](src/App.jsx)
	UIと状態管理の中心。
- [src/styles.css](src/styles.css)
	スマホ前提のカードUIと配色。
- [server/index.js](server/index.js)
	ローカル開発用の最小API。
- [vite.config.js](vite.config.js)
	開発サーバー設定と静的公開向けbase設定。
- [.github/workflows/deploy-pages.yml](.github/workflows/deploy-pages.yml)
	main pushでの自動デプロイ。

### 3-3. 実装で対応した変更履歴（要点）

- セル単位いいねからカード単位いいねへ変更
- いいねは取り消し可能へ変更
- 確認ボタンを廃止し、抽出条件を最終行到達へ変更
- コメント欄の下に いいねトグルを配置
- ページ遷移をさらに下へ配置
- 白ベース + 青アクセントの配色へ変更
- CSV平文貼り付けでの取り込みを追加

## 5. デプロイ（同一Wi-Fi不要で使う）

### 5-1. GitHub Pages公開

このプロジェクトは静的公開に対応済みです。
mainにpushすると、Actionsで自動デプロイされます。

手順

1. GitHubリポジトリを作成
2. mainにpush
3. GitHubの Pages 設定で Source を GitHub Actions にする
4. Actions完了後、公開URLへスマホでアクセス

公開URL例

- https://your-user-name.github.io/your-repo-name/

## 6. 利用手順（エンドユーザー向け）

1. 公開URLを開く
2. CSVをアップロード、またはCSV平文を貼り付けて読み込む
3. カードを読み、コメントを追加
4. いいねON/OFFを必要に応じて切替
5. 最後の行まで一度到達したらCSVをダウンロード

## 7. 品質確認チェックリスト

- CSV読み込み時に列崩れしない
- CSV平文貼り付けでも同じ結果で読み込める
- 1ページ1カードで表示される
- いいねがトグル動作する
- コメント追加と削除ができる
- 最終行到達前はダウンロードできない
- 出力CSVに追加列が入る
- スマホで押しやすいUIになっている

## 8. 用語解説（初心者向け）

- 要件定義
	作るもの・作らないもの・完成条件を決める工程。
- 非機能要件
	機能そのものではなく、使いやすさ・速度・保守性などの品質要件。
- UI
	画面の見た目や操作部品。
- UX
	使ったときの体験全体。
- 状態管理
	画面で変化するデータをプログラム内で保持・更新すること。
- トグル
	ON/OFFを切り替える操作。
- 平文
	ファイルではなく、テキストとして直接貼り付けた内容のこと。
- エクスポート
	アプリ内データを外部ファイルとして保存すること。
- JSON文字列
	データ構造を文字列化した形式。配列やオブジェクトを保存しやすい。
- ビルド
	公開用にアプリを最適化して出力する処理。
- デプロイ
	作ったアプリを公開環境で使えるように配置すること。
